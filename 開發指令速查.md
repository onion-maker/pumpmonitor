# 開發常用指令速查

## 一、ADB Logcat 除錯

### 基本用法
```powershell
# 清除舊 log 後只看 PumpMonitor 的訊息
adb logcat -c
adb logcat -s PumpMonitor

# 只看 JS console.log（WebView 輸出）
adb logcat -s chromium:V

# 同時看 Java + JS 兩種 log
adb logcat -s PumpMonitor chromium:V

# 只看錯誤層級以上
adb logcat -s PumpMonitor:E chromium:W

# 儲存到檔案（避免終端機爆掉）
adb logcat -s PumpMonitor chromium:V > debug_log.txt
```

### 依關鍵字過濾
```powershell
# 只看包含 "alarm" 或 "pump" 的訊息（不分大小寫）
adb logcat -s PumpMonitor | findstr /i "alarm pump"

# 只看包含 "check" 的訊息
adb logcat -s PumpMonitor | findstr /i "check"

# 保存後用更多關鍵字慢慢翻
adb logcat -s PumpMonitor chromium:V > debug_log.txt
notepad debug_log.txt
```

### 常見 Tag 對照
| Tag | 來源 | 內容 |
|-----|------|------|
| `PumpMonitor` | Java 背景服務 | API 檢查、警報觸發、排程 |
| `chromium` | WebView console.log | 前端 JS 除錯訊息 |
| `AppUpdate` | GitHub 更新檢查 | 版本比對、下載狀態 |

> **注意：** `debug_log.txt` 檔案很大（可能數十 MB），用完記得刪掉，也不要 commit 進 git。

### 前端 console.log 除錯輸出說明

程式碼中加了一些 `console.log`，用來診斷前端有沒有正確偵測到抽水機變化。當你發現 App 有異常時，可以用以下方式抓 log 給我看：

#### 抓 log 指令
```powershell
# 只抓前端 console.log（chromium tag）+ Java 背景服務（PumpMonitor tag）
adb logcat -c
adb logcat -s chromium:V PumpMonitor > debug_log.txt
```

讓 App 正常操作一陣子（或重現你遇到的問題），然後按 `Ctrl+C` 停止。把 `debug_log.txt` 傳給我就好。

#### 目前有加的 console.log 記錄

| 關鍵字 | 什麼時候出現 | 範例輸出 |
|--------|-------------|---------|
| `[checkAlarm]` + `啟動` | 前端偵測到抽水機從停止→運轉 | `[checkAlarm] 108 #3 啟動: 0→1` |
| `[checkAlarm]` + `停止` | 前端偵測到抽水機從運轉→停止 | `[checkAlarm] 112 #7 停止: 1→0` |

#### 解讀範例
```
[checkAlarm] 108 #3 啟動: 0→1
  ↑            ↑   ↑        ↑
  checkAlarm   站號 機組編號   狀態變化（0=停止, 1/2/3=運轉）
```

- **108** = 中山抽水站（對照 config/stations.ts）
- **#3** = 第 3 號抽水機
- **0→1** = 從停止變成運轉（閘控系統操作）

#### 常見狀況判斷

| log 輸出 | 代表什麼 | App 應該有反應嗎？ |
|----------|---------|------------------|
| 有 `[checkAlarm] #3 啟動` 的 log | 前端有偵測到變化 ✅ | 應該出現紅底警報橫幅 + 底部確認按鈕 |
| 沒有 `[checkAlarm]` 的 log | 前端沒偵測到變化 ❌ | App 不會有任何警報反應 |

如果 log 有顯示 pump 變化但 App 沒顯示警報 → 問題在前端顯示邏輯。
如果 log 完全沒 pump 變化記錄 → 問題在輪詢或比對邏輯。

## 二、Git Commit 流程

### 一般修改
```powershell
# 看改了哪些檔案
git status

# 看詳細改了什麼
git diff

# 全部加入 + commit
git add -A
git commit -m "標題：簡短描述修改內容"
```

### 修正上一個 commit（尚未 push 時）
```powershell
git add -A
git commit --amend --no-edit   # 不改訊息，只加檔案
# 或
git commit --amend -m "新的 commit 訊息"
```

> ⚠ `amend` 後只能用 `git push --force-with-lease`（推之前確認沒人動過）

### Commit 訊息格式建議
```
類型: 簡短說明（50 字以內）

類型:
  Feat:   新功能
  Fix:    修 bug
  UI:     畫面調整
  Refactor: 重構不改行為
  Docs:   文件
```

## 三、發佈新版流程

### 查看目前版本
```powershell
# 看最新 tag
git tag -l --sort=-v:refname | head -3

# 看最近 commit
git log --oneline -5
```

### 完整發佈步驟（以 v1.5.13 為例）

```powershell
# 0. 確保程式可編譯
npm run build

# 1. 看最新版號，確認下一版要打什麼 tag
git tag -l --sort=-v:refname | head -3

# 2. Push commit（一定要先，否則 CI 會抓到舊程式碼）
git push origin main

# 3. 如果 tag 已存在，先刪舊的
git push origin :refs/tags/v1.5.13
git tag -d v1.5.13

# 4. 打新 tag + push（觸發 CI 自動 build APK + Release）
git tag v1.5.13
git push origin main v1.5.13
```

> ⚠ **版號檢查要點：** 每次發佈前先用 `git tag -l --sort=-v:refname | head -3` 看最新版號，避免 tag 倒退覆蓋已發布的版本。如果已經 push 了舊 tag，就往上打下一號即可（例如 v1.5.11 → v1.5.13）。
```

### CI 自動做的事
- Build web（Vite）
- Sync Capacitor
- 自動填 versionCode＝日期、versionName＝tag 數字
- Build 簽名 APK
- 建立 GitHub Release + 上傳 APK

### 手機更新
Release 發布後，App 會自動檢查更新（開 App 時或背景每 24 小時檢查一次）→ 跳出更新提示 → 下載安裝。

### 如果 push 順序錯了（先 push tag 才 push commit）
```powershell
# 刪遠端 tag 重打
git push origin :refs/tags/v1.5.11
git tag -d v1.5.11
git tag v1.5.11
git push origin main v1.5.11
```

### 本機 Build APK（非必要，CI 已自動處理）
```powershell
npm run cap:build
```
APK 在 `android/app/build/outputs/apk/debug/app-debug.apk`
